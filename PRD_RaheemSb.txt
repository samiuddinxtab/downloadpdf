PRODUCT REQUIREMENTS DOCUMENT

Digital Islamic PDF Library

Version 5.0 — Final Repository Specification

---

1. PRODUCT VISION

A mobile-first digital library for curated Islamic articles in PDF format, optimized for knowledge organization, fast access, and long-term archival stability.

Attribute	Specification

Content Languages	Urdu, Arabic
Interface Language	Urdu (RTL)
Primary Distribution	Messaging platforms (shared links)
Repository Type	Institutional digital archive
Designated Community	Urdu and Arabic readers with basic Islamic scholarly literacy

> OAIS Note: Designated Community defines who the archive serves and what knowledge they bring to interpret the preserved information.

---

2. CORE OBJECTIVES

#	Objective

1	Provide structured archive of Islamic research articles
2	Enable instant PDF download from mobile devices
3	Maintain scholarly organization through metadata
4	Ensure long-term scalability and stability
5	Prevent uncontrolled redistribution via watermarking

6. METADATA MODEL

Descriptive Metadata (Required)

Field	Type	Normalization Rule

Title	Text	Fixed after publication
Author ID	Reference	Links to authority record
Category	Select	From controlled list
Tags	Select	From hierarchical vocabulary
Language	Select	Urdu / Arabic only
Synopsis	Text	Max 500 words
Pages	Number	Auto-extracted
File Size	Number	Auto-extracted
Upload Date	Date	Auto-generated
Unique ID	Text	Format: CAT-### (permanent)
PDF File	File	Immutable after upload
Slug	Text	Never changes after publish
Checksum	Hash	SHA-256 for integrity verification

Preservation Metadata (Required)

Field	Description

Creation Method	Original source type
Digitization Source	Scan / Digital native
Checksum Generation Date	When hash was created
Version History	All version records
Preservation Actions	Migrations, repairs, updates
Format Version	PDF standard version
Last Fixity Check	Date of last verification

Optional Citation Fields

Field	Description

Quran References	Structured verse citations
Hadith References	Structured hadith citations
Classical Sources	Book/title references

Metadata Export Format

Requirement	Specification

Export Format	JSON or XML
Export Scope	Full metadata record per article
Export Trigger	On-demand or scheduled
Purpose	System migration and portability

> Rule: Repositories live longer than software. Metadata must be portable.

13. PDF VERSION POLICY

Change Type	Action	Version Format

Minor Correction	Keep same ID	FIQH-017 v1.0 → v1.1
Major Revision	New version number	FIQH-017 v1.0 → v2.0
New Article	New unique ID	FIQH-018 v1.0

Archival Integrity Rules

Rule	Specification

Original PDF	Never overwritten
New Version	Stored as separate file
History	All versions retained
Checksum	Generated on upload, verified periodically

---

14. FILE FIXITY CHECKS

Verification Schedule

Frequency	Scope

Monthly	New files uploaded in period
Quarterly	Entire archive verification
Immediate	After any file migration
On-Demand	Before public access restoration

Fixity Check Process

Step	Action

1	Retrieve stored checksum from metadata
2	Generate new checksum from stored file
3	Compare checksums
4	Log result in preservation metadata
5	Alert if mismatch detected

24. BACKUP & DISASTER RECOVERY

Backup Schedule

Requirement	Specification

Frequency	Automated daily incremental
Full Backup	Weekly complete archive
Storage	Independent archive copy
Redundancy	Multiple storage locations

Archive Package Structure

archive-package/  
├── content/  
│   └── [all PDF files]  
├── metadata/  
│   └── [all metadata records]  
├── checksums/  
│   └── [all file hashes]  
└── manifest.json  
    └── [package inventory]

Disaster Recovery Scenario

Step	Action

1	Retrieve verified archive package from backup
2	Restore content files to new storage
3	Restore metadata records
4	Recalculate all checksums for verification
5	Rebuild search index from metadata
6	Validate fixity against manifest
7	Restore public access

> Purpose: Turns backup into actual disaster recovery capability.
